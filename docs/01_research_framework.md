# 01. Research Framework

## 1. Project Purpose

This project analyzes the monetization and live-ops structure of a typical casual / fishing-style mobile game.

The goal is to demonstrate the ability to connect:

- Core gameplay loop
- Currency and resource economy
- Resource production and consumption
- Paid bundle triggers
- Rewarded ad design
- Live-ops event rhythm
- User segmentation
- Retention and monetization metrics
- Commercialization risk
- Tuning recommendations

The project is built as a personal portfolio analysis. It does not use internal company data.

## 2. Applicable Roles

| Role | Relevant Ability Demonstrated |
|---|---|
| Game Operations | Live-ops rhythm, activity participation, retention and payment analysis |
| System Design | Core loop, currency systems, activity structures, player progression |
| Numerical Design | Resource production, consumption, scarcity, reward value, KPI interpretation |
| Monetization Design | Paid bundles, first purchase, battle pass, recharge rebate, event packs |
| Casual Game Design | Short-session loop, ad monetization, low-friction activities |
| User Growth / UA | LTV, ROAS, retention, payer conversion, user segmentation |
| Platform Operation | Event calendar, player lifecycle management, activity diagnosis |

## 3. Analysis Object

The project does not bind itself to one specific game.

It constructs a representative model based on common mechanics found in:

- Fishing arcade games
- Casual mobile games
- Mini games
- Board / card casual products
- Hybrid casual games
- Ad + IAP mixed monetization products

## 4. Excluded Scope

This project does not include:

- Real backend data
- Revenue prediction for a specific company
- Confidential product data
- Internal user cohorts
- Real payment funnel data
- Gambling or cash-out system design
- Claims about any specific company's performance

## 5. Methodology

The analysis uses the following method:

1. Observe player-visible systems  
2. Identify resource inputs and outputs  
3. Map player states to monetization triggers  
4. Build reasonable assumptions  
5. Construct simulated metrics  
6. Analyze KPI changes  
7. Identify design risks  
8. Propose tuning directions

## 6. Observable Signals and Possible Inferences

| Player-visible Content | Possible Design Inference |
|---|---|
| Login rewards | Retention habit formation |
| Daily tasks | Daily active behavior guidance |
| Room entry requirements | Resource gating and player segmentation |
| Cannon / ammo multiplier | Resource consumption speed |
| Bankruptcy popup | Resource shortage monetization |
| First purchase offer | First payer conversion |
| Daily / weekly packs | Payment habit and resource supplement |
| Event tokens | Short-cycle event economy |
| Leaderboard | Competitive monetization pressure |
| Battle pass | Mid-cycle retention and stable purchase |
| Rewarded ads | F2P safety valve and ad revenue |
| VIP / recharge tiers | Long-term payer identity and high-value retention |
